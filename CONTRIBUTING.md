# Contributing

This is a living guide. If a chapter is wrong, out of date, or missing a habit
that's been working for you, please improve it.

## How to propose a change

- For a small fix (a typo, a clearer sentence, a broken link), open a pull
  request directly.
- For a bigger change (a new chapter, a reworked section, a different
  recommendation), open an issue first so we can agree on the shape before you
  write it. That's the same "design before you build" loop the guide preaches —
  see `06-brainstorming-and-design-loops.md`.

## What good looks like here

The guide has a style. Keep new writing consistent with it:

- **One topic per chapter.** If your addition is really a new subject, it's a new
  chapter, not a bolt-on to an existing one. Update the table in `README.md` when
  you add one.
- **Hand-written prose.** Plain, practical, scannable. No `## What / ## How`
  scaffolding, no emoji headers, no filler. Short paragraphs and the occasional
  table — match what's already there.
- **Generic, always.** This repo is public. No company names, internal hosts,
  ticket or PR numbers, real credentials, or anything specific to a private
  codebase. Examples should make sense to any reader.
- **Show, don't assert.** Prefer a concrete snippet, command, or example prompt
  over a claim. If you recommend a tool, say what it's for and what it costs —
  the same honest lens as `07-evaluating-open-source.md` and
  `10-plugins-and-extensions.md`.
- **Stand on its own.** A chapter shouldn't depend on outside context a reader
  won't have. Cross-link sibling chapters by filename when it helps.

## Before you open the PR

- Read your change once more as if you were new to the topic — does it make sense
  cold?
- Check links and filenames resolve.
- Keep the PR description plain and specific: what you changed and why.
