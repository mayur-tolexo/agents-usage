# 04 — Pull requests

Let the agent handle the mechanics of git and GitHub — branching, committing,
opening the PR. Where you keep a hand on the wheel is the *description* and the
*push*.

## Let it do the plumbing

Asking the agent to run git and `gh` is one of the higher-leverage habits. It can:

- create a branch, stage, and commit with a sensible message
- pull, rebase, and resolve the easy conflicts
- open the PR and fill in the body
- run `git bisect` to find the commit that broke something (give it a test script
  so it can check each step)

A common and sensible split: **allow pull automatically, keep push behind a
prompt.** Pull is safe; push changes shared state. See chapter 02 for how to
encode that.

## Write commit messages the agent, review them yourself

You shouldn't be hand-typing commit messages — that's exactly the kind of thing
the agent is good at. But skim what it wrote. A good commit message says *what*
changed and *why* in plain language, and isn't padded with ceremony.

Two house rules worth enforcing (via memory, chapter 03):
- No AI co-author or "generated by" trailer.
- No individual names tagged into messages — keep them about the change.

## The PR description is the part that must read human

This is the one place where agent output most often gives itself away, and where
it matters most, because the description is what a reviewer reads first. Avoid the
tell-tale shape:

- ❌ `## What`, `## How`, `## Testing` scaffolding with a bullet under each
- ❌ aligned tables and emoji section headers
- ❌ vague "improved X and enhanced Y" filler

Aim instead for what a careful engineer would actually write:

- ✅ a short paragraph of plain prose: what this changes and why it's needed
- ✅ the one or two things a reviewer should look at closely
- ✅ real evidence — the actual command you ran and its actual output, pasted

> **A reasonable description**
>
> Connect tokens were being verified against the active key only, so a token
> signed with the previous key failed right after a rotation. This checks the
> token against all currently-valid keys instead.
>
> The tricky part is the rotation window — I added a test that signs with the old
> key and verifies after a rotation. Worth a close look at `verify.go`.
>
> ```
> $ go test ./auth/...
> ok   auth    0.412s
> ```

Note it explains the *why*, points the reviewer at the risky file, and pastes a
real test run instead of claiming "tests pass."

## Draft PRs are a verification tool

Opening the PR as a **draft** is one of the best ways to check the agent's work:
you get the full diff in GitHub's review UI, can leave comments, and nothing is
"ready" until you mark it so. Generate the draft, review it like you'd review a
colleague's, then promote it. (A visual git client works the same way for the
pre-commit stage — eyeball the diff before it becomes a commit.)

## The shape of a good PR flow

1. Branch off the default branch.
2. Make the change in small commits.
3. Run the build and tests; paste the real output.
4. Open a **draft** PR with a hand-written description.
5. You review the diff.
6. You (not the agent) mark it ready and merge.

Steps 1–4 the agent does; steps 5–6 are yours. That division keeps velocity high
and keeps the irreversible action behind a human.
